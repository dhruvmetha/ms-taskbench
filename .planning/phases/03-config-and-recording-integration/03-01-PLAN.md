  <action>
  1. In ps_bed/config.py, add `num_cubes: int = 3` to the EnvConfig dataclass, placed after `max_episode_steps` and before `record_video`.

  2. In configs/default.yaml, add `num_cubes: 3` to the `env:` section, placed after `max_episode_steps` and before `record_video` to mirror the dataclass field order.

  3. In ps_bed/env.py, add a helper to build extra kwargs for gym.make() based on which env is being created. The key logic: only pass `num_cubes` when the env actually accepts it. Use a simple string check -- if `cfg.env_id` contains "StackNCube", include `num_cubes=cfg.num_cubes` in the kwargs dict; otherwise omit it entirely. This prevents crashing StackCube-v1 or StackCubeDistractor-v1 with an unexpected kwarg.

  Implementation approach for env.py:
  ```python
  def _extra_env_kwargs(cfg: EnvConfig) -> dict:
      """Build env-specific kwargs based on env_id."""
      kwargs = {}
      if "StackNCube" in cfg.env_id:
          kwargs["num_cubes"] = cfg.num_cubes
      return kwargs
  ```

  4. In make_env(), unpack `**_extra_env_kwargs(cfg)` into the gym.make() call (after the existing kwargs).

  5. In make_single_env(), unpack `**_extra_env_kwargs(cfg)` into the gym.make() call (after the existing kwargs).

  This ensures:
  - `python -m ps_bed.run` (defaults: StackCube-v1, random) -- no num_cubes passed, works fine
  - `python -m ps_bed.run run.policy=pick_place` (StackCube-v1) -- no num_cubes passed, works fine
  - `python -m ps_bed.run env.env_id=StackNCube-v1 env.num_cubes=5` -- num_cubes=5 passed, works fine
  </action>
