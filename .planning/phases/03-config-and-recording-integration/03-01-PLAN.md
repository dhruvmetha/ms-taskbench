  <files>ps_bed/config.py, ps_bed/env.py, configs/default.yaml</files>
  <action>
  1. In ps_bed/config.py, add `num_cubes: int = 3` to the EnvConfig dataclass, placed after `max_episode_steps` and before `record_video`.

  2. In configs/default.yaml, add `num_cubes: 3` to the `env:` section, placed after `max_episode_steps` and before `record_video` to mirror the dataclass field order.

  3. In ps_bed/env.py make_env(), add `num_cubes=cfg.num_cubes` to the gym.make() call. This enables vectorized envs to also use StackNCube-v1 with configurable cube count.

  4. In ps_bed/env.py make_single_env(), add `num_cubes=cfg.num_cubes` to the gym.make() call. This is the critical path for motion planner usage.

  Note: gym.make() forwards unknown kwargs to the env constructor. For envs that don't accept num_cubes (e.g., StackCube-v1), this will cause a TypeError. This is acceptable -- the user should use StackNCube-v1 when specifying num_cubes. The default env_id remains StackCube-v1 and default num_cubes=3, but the user is expected to set env.env_id=StackNCube-v1 when using stack_n policy.

  IMPORTANT: Do NOT auto-switch env_id based on policy. Keep env_id independent per research recommendation -- explicit is better than implicit.
  </action>
